# API Response Standardization
 
All API responses — regardless of language, framework, or service — MUST follow this standard structure. This ensures consistency across all FKS services and enables clients to handle responses uniformly.
 
---
 
## Standard Response Envelope
 
Every API response wraps its payload in a two-field envelope:
 
```json
{
  "meta": { ... },
  "data": { ... }
}
```
 
### `meta` — Response Metadata
 
Always present. Contains status, message, and tracing information.
 
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `correlationId` | string | Yes | Request tracing ID. Echo from `X-Correlation-ID` header, or generate a UUID if absent |
| `status` | boolean | Yes | `true` = success, `false` = failure |
| `code` | integer | Yes | HTTP-aligned status code (see [Status Codes](#status-codes)) |
| `message` | string | Yes | Human-readable result message |
| `exceptionMessage` | string | No | Technical error detail. Empty string on success, error detail on failure |
| `pagination` | object | No | Present only on paginated list responses |
 
### `data` — Response Payload
 
| Response Type | Shape | When to use |
|---------------|-------|-------------|
| Single record | `{ "record": { ... } }` | GET by ID, Create, Update, Delete |
| List | `{ "records": [ ... ] }` | Paginated list endpoints |
| Empty | `null` | Mutation operations (Create/Update/Delete) that return no data |
 
---
 
## Pagination Object
 
Present in `meta.pagination` for list responses only.
 
```json
{
  "page": 1,
  "limit": 10,
  "totalRecord": 42,
  "totalPage": 5,
  "nextPage": true,
  "previousPage": false,
  "nextPageURL": "https://api.example.com/Resource/GetList?page=2&limit=10&...",
  "previousPageURL": "",
  "filterColumn": "name",
  "searchTerm": "john",
  "orderBy": "{'CreatedAt':'DESC'}"
}
```

## Status Codes
 
`meta.code` mirrors standard HTTP semantics even though the transport layer always returns `200` (see [Rules](#rules)). Use the most specific code that applies — don't collapse everything into `400`/`500`.
 
| Code | Meaning | Typical use |
|------|---------|--------------|
| `200` | OK | Successful GET, Update, Delete |
| `201` | Created | Successful Create (resource persisted) |
| `400` | Bad Request | Validation failure, malformed input, business rule violation |
| `401` | Unauthorized | Missing, invalid, or expired token |
| `403` | Forbidden | Authenticated but not permitted to access the resource |
| `404` | Not Found | Requested record/resource does not exist |
| `409` | Conflict | Duplicate resource, unique constraint violation, state conflict |
| `422` | Unprocessable Entity | Semantically invalid payload (passes schema, fails domain rules) |
| `429` | Too Many Requests | Rate limit exceeded |
| `500` | Internal Server Error | Unhandled exception / server fault |
| `503` | Service Unavailable | Downstream dependency unavailable (DB, queue, external API) |
 
`meta.status` is always the boolean shorthand: `true` only when `code` is `200` or `201`; `false` for every other code.

## Standard Request Headers
 
All requests to FKS services MUST include:
 
| Header | Required | Description |
|--------|----------|-------------|
| `Authorization` | Yes | `Bearer {jwt_token}` |
| `X-Correlation-ID` | Recommended | UUID for request tracing. Generated by gateway or client. Echoed back in `meta.correlationId` |
| `Content-Type` | Yes (POST/PUT) | `application/json` |
 
---
 
## Pagination Query Parameters
 
All list endpoints accept these standard query parameters:
 
| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `page` | integer | 1 | Current page number (1-based) |
| `limit` | integer | 10 | Page size. Use `1000` to request all records (maps to `int.MaxValue`) |
| `filterColumn` | string | `""` | Column name to filter on |
| `searchTerm` | string | `""` | Search value for the filter column |
| `orderBy` | string | `"{'Id':'ASC'}"` | JSON-style sort expression, e.g. `{'CreatedAt':'DESC'}` |

## Standard Message Constants
 
Use these standard messages consistently across all services:
 
| Constant | Value |
|----------|-------|
| Create success | `"Data has been created"` |
| Update success | `"Data has been updated"` |
| Delete success | `"Data has been deleted"` |
| Create failed | `"Create data failed!"` |
| Update failed | `"Update data failed!"` |
| Delete failed | `"Delete data failed!"` |
| Data found | `"Data found!"` |
| Data not found | `"Data not found!"` |
| Invalid input | `"Input data not found or invalid!"` |
| Unauthorized | `"Token invalid."` |
| Forbidden | `"You do not have permission to access this resource."` |
| Conflict | `"Record already exists."` |
| Rate limited | `"Too many requests. Please try again later."` |
| Server error | `"Internal server error"` |
| Service unavailable | `"Service temporarily unavailable"` |

## TypeScript Types
 
Reference types for services/clients written in TypeScript. Place these in a shared package (e.g. `@fks/api-types`) so every service and frontend consumes the same contract.
 
```typescript
/** HTTP-aligned status codes used in `meta.code`. */
export type ApiStatusCode =
  | 200 // OK
  | 201 // Created
  | 400 // Bad Request
  | 401 // Unauthorized
  | 403 // Forbidden
  | 404 // Not Found
  | 409 // Conflict
  | 422 // Unprocessable Entity
  | 429 // Too Many Requests
  | 500 // Internal Server Error
  | 503; // Service Unavailable
 
/** Sort direction used inside the `orderBy` expression. */
export type SortDirection = "ASC" | "DESC";
 
/** Pagination metadata attached to list responses. */
export interface ApiPagination {
  page: number;
  limit: number;
  totalRecord: number;
  totalPage: number;
  nextPage: boolean;
  previousPage: boolean;
  nextPageURL: string;
  previousPageURL: string;
  filterColumn?: string;
  searchTerm?: string;
  orderBy?: string;
}
 
/** The `meta` block present on every API response. */
export interface ApiMeta {
  correlationId: string;
  status: boolean;
  code: ApiStatusCode;
  message: string;
  exceptionMessage: string;
  pagination?: ApiPagination;
}
 
/** `data` shape for a single-record response. */
export interface ApiRecordData<T> {
  record: T;
}
 
/** `data` shape for a list/paginated response. */
export interface ApiListData<T> {
  records: T[];
}
 
/** Full envelope for an endpoint returning a single record. */
export interface ApiRecordResponse<T> {
  meta: ApiMeta;
  data: ApiRecordData<T> | null;
}
 
/** Full envelope for an endpoint returning a list. */
export interface ApiListResponse<T> {
  meta: ApiMeta;
  data: ApiListData<T> | null;
}
 
/** Full envelope for a mutation endpoint that returns no payload. */
export interface ApiMutationResponse {
  meta: ApiMeta;
  data: null;
}
 
/** Union covering any FKS API response shape — useful for generic HTTP client wrappers. */
export type ApiResponse<T = unknown> =
  | ApiRecordResponse<T>
  | ApiListResponse<T>
  | ApiMutationResponse;
 
/** Standard pagination query parameters accepted by list endpoints. */
export interface ApiListQuery {
  page?: number;
  limit?: number;
  filterColumn?: string;
  searchTerm?: string;
  orderBy?: string; // e.g. "{'CreatedAt':'DESC'}"
}
 
/** Type guard: narrows an ApiResponse to its success/failure state via `meta.status`. */
export function isApiSuccess<T>(
  res: ApiResponse<T>
): res is ApiRecordResponse<T> | ApiListResponse<T> {
  return res.meta.status === true;
}
 
/** Type guard: narrows `data` to the single-record shape. */
export function isRecordData<T>(
  data: ApiRecordData<T> | ApiListData<T> | null
): data is ApiRecordData<T> {
  return data !== null && "record" in data;
}
 
/** Type guard: narrows `data` to the list shape. */
export function isListData<T>(
  data: ApiRecordData<T> | ApiListData<T> | null
): data is ApiListData<T> {
  return data !== null && "records" in data;
}
```

## Rules
 
1. **Always return HTTP 200** at the transport level — use `meta.code` and `meta.status` to communicate the actual result. Never return HTTP 4xx/5xx directly from business logic.
2. **Never expose raw exception stack traces** to clients in production. Put technical details in `exceptionMessage` only for debugging; mask in production if sensitive.
3. **Always echo `X-Correlation-ID`** from the request header into `meta.correlationId`. If the header is absent, generate a new UUID.
4. **Pagination URLs** must be fully qualified absolute URLs, not relative paths.
5. **Empty `data`** on mutations is intentional — do not return the mutated entity unless explicitly required by the endpoint contract.
6. **Use the most specific `meta.code`** available (e.g. `404` for a missing record, not a generic `400`) so clients can branch on it without parsing `message`.
7. **TypeScript services/clients** must import the shared `ApiResponse`/`ApiMeta`/etc. types rather than redeclaring the envelope shape locally, to avoid drift between services.
